1. Visão Geral do Produto

Problema que Resolve

Grupos de esportes e atividades recorrentes dependem hoje de listas manuais no WhatsApp, planilhas desatualizadas e cobrança informal de PIX. Sem controle real de presença, fila de espera ou histórico financeiro.

!
Exemplo real: Carlos administra o "Futebol Quinta São Luís" com 10 jogadores fixos e 5 na lista de espera. Toda quarta-feira posta a lista no WhatsApp, edita manualmente quem confirmou, cobra PIX no privado, e na quinta sempre faltam 2 jogadores — Pedro saiu sem avisar, Rafael ficou esperando ser chamado e não foi.

3.2 Fila de Espera — Regras Completas

1
Rafael (mensalista) cancela presença no evento de quinta.
2
Sistema identifica André como primeiro da fila (posição 1).
3
Push notification: "🎉 Uma vaga abriu! Você tem 30 minutos para confirmar."
4a
André confirma em 15min → Status = CONFIRMED. Thiago sobe para posição 1.
4b
André não responde → vai para o final da fila. Thiago é notificado.

3.3 Mensalistas — Ciclo Completo

Dia  1 do mês → Cobrança gerada (R$ 60,00) + notificação
Dia  5 (venc.) → Pago: ativo ✅ | Não pago: aviso de inadimplência ⚠️
Dia 10 (grace) → Vaga suspensa se não pago. Organizador notificado.

Por evento (ex: toda quinta-feira):
  Dom 18h → Sistema cria evento para quinta
  Seg 08h → Notificação para mensalistas confirmarem
  Ter 22h → Prazo de confirmação dos mensalistas
  Qua 08h → Vagas não confirmadas → liberadas para fila de espera
  Qua 12h → Notificação para fila de espera
  Qui 19h → Lembrete: 1h antes do evento
  Qui 21h30 → Evento encerrado → organizador registra presença
  Sex 09h → Cobrança avulso gerada

// Server Action (preferência sobre API Routes para mutações)
'use server'

export async function confirmParticipation(input: { eventId: string; userId: string }) {
  const supabase = await createServerClient()
  const parsed = confirmParticipationSchema.parse(input)

  const { data: event } = await supabase
    .from('events')
    .select('max_participants, participant_count')
    .eq('id', parsed.eventId)
    .single()

  if (event.participant_count >= event.max_participants) {
    return { error: 'NO_SLOTS_AVAILABLE' }
  }

  await supabase.from('event_participants').upsert({
    event_id: parsed.eventId,
    user_id: parsed.userId,
    status: 'confirmed',
    confirmed_at: new Date().toISOString(),
  })

  revalidatePath(`/groups/[groupId]/events/${parsed.eventId}`)
  return { success: true }
}
// Realtime hook — lista de presença ao vivo
export function useEventRealtime(eventId: string) {
  const supabase = useSupabaseClient()

  useEffect(() => {
    const channel = supabase
      .channel(`event:${eventId}`)
      .on('postgres_changes', {
        event: '*', schema: 'public',
        table: 'event_participants',
        filter: `event_id=eq.${eventId}`,
      }, (payload) => handleParticipantChange(payload))
      .subscribe()

    return () => { supabase.removeChannel(channel) }
  }, [eventId])
}

Triggers Principais

CREATE TRIGGER trg_update_participant_count
  AFTER INSERT OR UPDATE OR DELETE ON event_participants
  FOR EACH ROW EXECUTE FUNCTION update_event_participant_count();

-- Aciona fila quando participante cancela
CREATE TRIGGER trg_notify_waitlist
  AFTER UPDATE ON event_participants
  FOR EACH ROW EXECUTE FUNCTION notify_waitlist_on_cancel();

-- Reordena posições da fila quando alguém sai

Cron Jobs (pg_cron)

-- Gerar eventos recorrentes (todo dia à meia-noite)
'0 0 * * *' → Edge Function: generate-recurring-events

-- Verificar prazos de mensalistas (a cada hora)
'0 * * * *' → Edge Function: check-monthly-deadlines

-- Expirar fila de espera sem resposta (a cada 5 min)
'*/5 * * * *' → UPDATE waitlists SET status='expired' WHERE expires_at < now()

-- Gerar cobranças mensais (dia 1, 8h)
'0 8 1 * *' → Edge Function: generate-monthly-charges

-- Marcar inadimplentes (todo dia 9h)
'0 9 * * *' → UPDATE payments SET status='overdue' WHERE due_date < CURRENT_DATE
